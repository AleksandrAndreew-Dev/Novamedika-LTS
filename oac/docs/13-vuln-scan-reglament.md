## 1. ОБЩИЕ ПОЛОЖЕНИЯ

1.1. Настоящий регламент определяет порядок сканирования уязвимостей и тестирования на проникновение в ИС NovaMedika2.

1.2. Регламент разработан в соответствии с требованиями приказа ОАЦ №66 (приложение 3, пункты 7.17, 7.18).

## 2. ИНСТРУМЕНТЫ СКАНИРОВАНИЯ

### 2.1. Trivy — Сканирование Docker-образов

| Параметр | Значение |
|----------|----------|
| Образ | `aquasec/trivy:latest` |
| Объект сканирования | Все Docker-образы (backend, frontend, postgres, redis, traefik) |
| Тип проверки | Уязвимости (CVE) + секреты |
| Severity | HIGH, CRITICAL |
| Формат отчёта | Таблица (текст) |
| Периодичность | При каждом обновлении образа |

### 2.2. Nikto — Сканирование веб-приложений

| Параметр | Значение |
|----------|----------|
| Образ | `bitnami/nikto:latest` |
| Объект сканирования | api.spravka.novamedika.com, spravka.novamedika.com |
| Тип проверки | Уязвимости веб-сервера, CGI, XSS, SQL-инъекции |
| Периодичность | Ежеквартально |

### 2.3. Nmap — Сетевое сканирование

| Параметр | Значение |
|----------|----------|
| Образ | `instrumentisto/nmap:latest` |
| Объект сканирования | spravka.novamedika.com (порты 80, 443, 22, 8000) |
| Тип проверки | Открытые порты, версии сервисов, известные уязвимости |
| Периодичность | Ежеквартально |

### 2.4. SSLyze — Проверка TLS-конфигурации

| Параметр | Значение |
|----------|----------|
| Образ | `nablac0d3/sslyze:latest` |
| Объект сканирования | spravka.novamedika.com |
| Тип проверки | TLS-сертификаты, протоколы, шифры |
| Периодичность | При обновлении сертификата + ежеквартально |

## 3. ПОРЯДОК СКАНИРОВАНИЯ

### 3.1. Автоматическое сканирование (CI/CD)

**При каждом деплое:**
```yaml
# .github/workflows/deploy.yml
- name: Trivy Scan
  run: |
    trivy image --severity HIGH,CRITICAL \
      ghcr.io/aleksandrandreew-dev/novamedika-lts/backend:latest
```

### 3.2. Плановое сканирование

**Скрипт:** `agent/oac/vuln-scan.sh`

```bash
# Запуск всех сканеров
bash agent/oac/vuln-scan.sh all

# Только backend
bash agent/oac/vuln-scan.sh backend

# Только frontend
bash agent/oac/vuln-scan.sh frontend
```

**Расписание:** ежеквартально, 1-й рабочий день квартала

### 3.3. Тестирование на проникновение

**Периодичность:** ежегодно

**Методы:**
- Сетевой сканер (Nmap)
- Сканер уязвимостей веб-приложений (Nikto)
- Ручное тестирование (при необходимости — через специализированную организацию)

## 4. КРИТЕРИИ ОЦЕНКИ РЕЗУЛЬТАТОВ

### 4.1. Severity-уровни

| Уровень | Описание | Действие |
|---------|----------|----------|
| CRITICAL | Уязвимость с высоким риском эксплуатации | Устранить в течение 24 часов |
| HIGH | Уязвимость с потенциальным риском | Устранить в течение 7 дней |
| MEDIUM | Уязвимость с низким риском | Устранить в течение 30 дней |
| LOW | Информационное замечание | Устранить при возможности |

### 4.2. Допустимые риски

| Компонент | CRITICAL | HIGH | MEDIUM |
|-----------|----------|------|--------|
| Backend | 0 | 0 | ≤ 3 |
| Frontend | 0 | 0 | ≤ 5 |
| Infrastructure | 0 | 0 | ≤ 2 |
| TLS | 0 | 0 | 0 |

## 5. ОТЧЁТНОСТЬ

### 5.1. Состав отчёта

Отчёт о сканировании (`agent/oac/vuln-reports/YYYYMMDD_HHMMSS/`):

| Файл | Описание |
|------|----------|
| `trivy-*.txt` | Результаты Trivy для каждого образа |
| `nikto-api.txt` | Результаты Nikto для API |
| `nikto-frontend.txt` | Результаты Nikto для Frontend |
| `nmap-full.txt` | Результаты Nmap |
| `sslyze.json` | Результаты SSLyze |

### 5.2. Хранение отчётов

- Расположение: `agent/oac/vuln-reports/`
- Срок хранения: бессрочно (для аудита)
- Доступ: ограничен (root, ответственный за защиту ИБ)

### 5.3. Периодический отчёт руководителю

**Периодичность:** ежеквартально

**Состав:**
- Количество уязвимостей по уровням (CRITICAL/HIGH/MEDIUM/LOW)
- Динамика (по сравнению с предыдущим кварталом)
- Выполненные исправления
- Рекомендации

---

## 6. ПОРЯДОК УСТРАНЕНИЯ УЯЗВИМОСТЕЙ

### 6.1. CRITICAL (24 часа)

1. Немедленно уведомить администратора
2. Оценить риск эксплуатации
3. Применить временное решение (workaround)
4. Обновить компонент/исправить код
5. Пересканировать для подтверждения исправления
6. Зафиксировать в журнале

### 6.2. HIGH (7 дней)

1. Запланировать исправление в ближайшем спринте
2. Обновить компонент/исправить код
3. Пересканировать
4. Зафиксировать в журнале

### 6.3. MEDIUM (30 дней)

1. Запланировать исправление
2. Исправить при обновлении компонента
3. Пересканировать
