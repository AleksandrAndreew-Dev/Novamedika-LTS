1.1. Настоящий регламент определяет порядок мониторинга событий информационной безопасности (далее — мониторинг ИБ) в ИС NovaMedika2.

## 3. ПЕРИОДИЧНОСТЬ И СПОСОБ МОНИТОРИНГА

### 3.1. Ежедневный мониторинг

**Ответственный:** Администратор ИС

**Время:** 09:00 (UTC+3)

**Состав:**
1. Просмотр дашборда Grafana (grafana.novamedika.com)
2. Проверка панели «Ошибки (ERROR/WARN)»
3. Проверка панели «Аутентификация — неудачные попытки»
4. Проверка панели «События безопасности»
5. Фиксация аномалий в журнале

**Действия при обнаружении:**
- WARN: зафиксировать в журнале, наблюдать
- HIGH: расследовать в течение 4 часов
- CRITICAL: немедленное реагирование, информирование ответственного за защиту ИБ

### 3.2. Еженедельный анализ

**Ответственный:** Ответственный за защиту информации

**Время:** Понедельник 10:00 (UTC+3)

**Состав:**
1. Анализ логов за предыдущую неделю
2. Подсчёт количества событий по уровням
3. Выявление трендов (рост/снижение аномалий)
4. Проверка выполнения требований ОАЦ
5. Формирование краткого отчёта

**Отчёт включает:**
- Количество событий аутентификации (успешных/неудачных)
- Количество ошибок 5xx
- Количество подозрительных запросов
- Выявленные инциденты и принятые меры
- Рекомендации по улучшению

### 3.3. Ежемесячный аудит

**Ответственный:** Ответственный за защиту информации

**Состав:**
1. Полный анализ логов за месяц
2. Проверка целостности архивных логов
3. Проверка работоспособности Loki/Promtail
4. Проверка объёма хранимых логов (≥ 365 дней)
5. Обновление регламента (при необходимости)

## 4. ХРАНЕНИЕ ЛОГОВ

### 4.1. Loki — горячее хранение

| Параметр | Значение |
|----------|----------|
| Срок хранения | 395 дней (~13 месяцев) |
| Объём | Зависит от нагрузки (ориентировочно 1-2 GB/месяц) |
| Доступ | Grafana (grafana.novamedika.com) |

### 4.2. Архив — холодное хранение

| Параметр | Значение |
|----------|----------|
| Срок хранения | 395 дней |
| Формат | gzip архивы по контейнерам и датам |
| Расположение | Docker volume `log_archive` → `./log_archive` |
| Периодичность архивирования | Каждые 6 часов |

### 4.3. Резервное копирование логов

- Архивные логи копируются на внешний носитель еженедельно
- Копия хранится отдельно от основной инфраструктуры

## 5. ЖУРНАЛ РЕГИСТРАЦИИ СОБЫТИЙ ИБ

### 5.1. Форма записи в журнале

| Поле | Описание |
|------|----------|
| Дата и время | YYYY-MM-DD HH:MM:SS (UTC+3) |
| Тип события | Аутентификация / Доступ / Ошибка / Подозрительная |
| Уровень | INFO / WARN / HIGH / CRITICAL |
| Источник | Компонент (backend, traefik, postgres, redis) |
| Описание | Краткое описание события |
| Принятые меры | Что сделано для реагирования |
| Ответственный | ФИО |

### 5.2. Ведение журнала

- Журнал ведётся в электронной форме (файл `oac/logs/security-log.xlsx`)
- Записи вносятся не позднее следующего рабочего дня после обнаружения
- Журнал хранится бессрочно

## 6. РЕАГИРОВАНИЕ НА ИНЦИДЕНТЫ

### 6.1. Уровни инцидентов

| Уровень | Описание | Время реакции | Эскалация |
|---------|----------|--------------|-----------|
| INFO | Информационное событие | — | Не требуется |
| WARN | Потенциальная угроза | 24 часа | Администратор |
| HIGH | Подтверждённая угроза | 4 часа | Ответственный за защиту ИБ |
| CRITICAL | Инцидент безопасности | 1 час | Руководитель + ОАЦ (в течение 24 часов) |

### 6.2. Порядок реагирования

**CRITICAL:**
1. Немедленно устранить угрозу (блокировка IP, отключение доступа)
2. Сохранить логи для расследования
3. Уведомить ответственного за защиту ИБ
4. В течение 1 часа — уведомить руководителя
5. В течение 24 часов — уведомить ОАЦ (если требуется по закону)
6. Провести расследование в течение 5 рабочих дней
7. Принять меры по недопущению повторения
