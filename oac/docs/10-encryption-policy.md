## 1. ОБЩИЕ ПОЛОЖЕНИЯ

1.1. Настоящий регламент определяет порядок генерации, хранения, использования и ротации криптографических ключей в ИС NovaMedika2.

## 2. КРИПТОГРАФИЧЕСКИЕ КЛЮЧИ

### 2.1. Перечень ключей

| Ключ | Алгоритм | Длина | Назначение | Где хранится |
|------|----------|-------|-----------|-------------|
| ENCRYPTION_KEY | AES-256 (pgcrypto) | 32 байта | Шифрование ПД в БД | PostgreSQL GUC + .env |
| FERNET_KEY | Fernet (AES-128-CBC) | 32 байта Base64 | Шифрование API-токенов аптек | .env |
| SECRET_KEY | HMAC-SHA256 | 32+ байта | JWT подписи | .env |
| REDIS_PASSWORD | — | 16+ символов | Аутентификация Redis | .env |
| POSTGRES_PASSWORD | — | 16+ символов | Аутентификация PostgreSQL | .env |
| SSL_EMAIL | — | — | TLS-сертификаты Let's Encrypt | .env |

### 2.2. Требования к ключам

| Ключ | Генерация | Сложность | Ротация |
|------|----------|-----------|---------|
| ENCRYPTION_KEY | `openssl rand -base64 32` | 256 бит | 1 год |
| FERNET_KEY | `python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"` | 128 бит | 1 год |
| SECRET_KEY | `openssl rand -base64 64` | 512 бит | 6 месяцев |
| REDIS_PASSWORD | Генератор паролей | 16+ символов, [A-Za-z0-9!@#$%] | 6 месяцев |
| POSTGRES_PASSWORD | Генератор паролей | 16+ символов, [A-Za-z0-9!@#$%] | 6 месяцев |

## 3. ПОРЯДОК ХРАНЕНИЯ КЛЮЧЕЙ

### 3.1. Production

| Хранилище | Какие ключи | Доступ |
|-----------|------------|--------|
| .env на сервере | Все ключи | root, novamedika (владелец), chmod 600 |
| PostgreSQL GUC (app.encryption_key) | ENCRYPTION_KEY | Только DBA, SUPERUSER |
| GitHub Secrets (CI/CD) | Все ключи (для деплоя) | Только администраторы репозитория |

### 3.2. Требования к защите

- Файл .env: `chmod 600`, владелец — `novamedika:novamedika`
- Ключи НЕ коммитятся в git
- Ключи НЕ передаются по незащищённым каналам
- При компрометации — немедленная ротация

## 4. ПОРЯДОК РОТАЦИИ КЛЮЧЕЙ

### 4.1. Плановая ротация

| Ключ | Периодичность | Ответственный |
|------|--------------|--------------|
| ENCRYPTION_KEY | 1 год | Администратор БД |
| FERNET_KEY | 1 год | Администратор backend |
| SECRET_KEY | 6 месяцев | Администратор backend |
| REDIS_PASSWORD | 6 месяцев | Администратор инфраструктуры |
| POSTGRES_PASSWORD | 6 месяцев | Администратор БД |

### 4.2. Процедура ротации ENCRYPTION_KEY

```bash
# 1. Генерируем новый ключ
NEW_KEY=$(openssl rand -base64 32)

# 2. В production: обновляем .env
sed -i "s/ENCRYPTION_KEY=.*/ENCRYPTION_KEY=${NEW_KEY}/" /opt/novamedika-prod/.env

# 3. Обновляем GUC в PostgreSQL
docker exec postgres-prod psql -U postgres -c \
  "ALTER DATABASE novamedika SET app.encryption_key = '${NEW_KEY}';"

# 4. Перезапускаем PostgreSQL
docker restart postgres-prod

# 5. Решифруем данные (скрипт re-encrypt)
docker exec postgres-prod psql -U postgres -d novamedika -f /tmp/re-encrypt.sql

# 6. Проверяем работоспособность
docker exec postgres-prod psql -U postgres -d novamedika -c \
  "SELECT count(*) FROM v_booking_orders_decrypted;"
```

### 4.3. Процедура ротации при компрометации

**Срок:** немедленно (в течение 1 часа)

1. Сгенерировать новый ключ
2. Обновить все хранилища
3. Перешифровать данные
4. Отозвать старый ключ
5. Зафиксировать в журнале инцидентов
6. Уведомить ответственного за защиту ИБ
