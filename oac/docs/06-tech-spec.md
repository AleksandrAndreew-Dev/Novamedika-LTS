**Особенности применения согласно Приказу ОАЦ № 259:**

1. **Пункт 5.1 (обязательный для 3-ин):** Шифрование при передаче реализовано через HTTPS/TLS 1.3. Если данные получены от субъекта без криптозащиты (например, через Telegram Bot API), их возврат тому же субъекту может осуществляться без дополнительного шифрования (пункт 19¹ Положения).

2. **Пункт 5.2 (условный для 3-ин):** Шифрование при хранении реализовано для наиболее чувствительных персональных данных (телефоны, Telegram ID) согласно принципу пропорциональности. Используется расширение PostgreSQL pgcrypto с алгоритмом AES-256-CBC. Полное шифрование БД (LUKS/dm-crypt) или шифрование на уровне приложения для всех ПД может быть реализовано при необходимости повышения уровня защищенности.

3. **Управление ключами шифрования:** Согласно Регламенту управления ключами шифрования (документ 10), ключи ENCRYPTION_KEY и FERNET_KEY ротируются ежегодно, хранятся в защищенном .env файле (chmod 600) и PostgreSQL GUC параметрах. Процедура ротации документирована и протестирована.

4. **⚠️ ВАЖНОЕ ПРИМЕЧАНИЕ О СЕРТИФИКАЦИИ:** Согласно п.13 Положения о технической и криптографической защите информации (Указ №449) и п.7 ст.28 Закона №455-З, средства технической и криптографической защиты должны иметь сертификат соответствия Национальной системы подтверждения соответствия или положительное экспертное заключение ОАЦ. В настоящее время используемые библиотеки (cryptography/Fernet, PyJWT, pgcrypto) не имеют таких сертификатов. Данный вопрос находится в стадии проработки — рассматриваются варианты замены на сертифицированные аналоги, получения экспертного заключения или согласования компенсирующих мер с ОАЦ.

## 4. ТРЕБОВАНИЯ К СРЕДСТВАМ КРИПТОГРАФИЧЕСКОЙ ЗАЩИТЫ ИНФОРМАЦИИ

### 4.1. TLS-шифрование

| Параметр | Требование |
|----------|-----------|
| Протокол | TLS 1.2+ (предпочтительно TLS 1.3) |
| Сертификаты | Let's Encrypt (ACME), автоматическое обновление |
| Алгоритм ключей | ECDSA или RSA 2048+ бит |
| Срок действия сертификата | 90 дней (автоматическое обновление) |

### 4.2. JWT-аутентификация

| Параметр | Требование |
|----------|-----------|
| Алгоритм подписи | HMAC-SHA256 |
| Access token | 30 минут |
| Refresh token | 7 дней с отзыванием в БД |
| Секретный ключ | 32+ байта, генерация |

### 4.3. Fernet-шифрование

| Параметр | Требование |
|----------|-----------|
| Алгоритм | AES-128-CBC + HMAC-SHA256 |
| Применение | Шифрование API-токенов аптек в БД |
| Ключ | 32 байта Base64, хранение в .env |

### 4.4. Хэширование паролей

| Параметр | Требование |
|----------|-----------|
| Алгоритм | bcrypt или argon2 |
| Соль | Автоматическая генерация |

### 5.3. Утверждение локальных актов

Утверждение проектов локальных правовых актов по вопросам применения системы защиты с учётом результатов внедрения.

## 6. ТРЕБОВАНИЯ К АТТЕСТАЦИИ СИСТЕМЫ ЗАЩИТЫ

### 6.1. Общие требования

Аттестация системы защиты информации проводится в соответствии с Положением о порядке аттестации систем защиты информации (приказ ОАЦ №66).

### 6.2. Состав аттестационных мероприятий

1. Разработка программы и методики аттестации
2. Проверка правильности отнесения ИС к классу 3-ин
3. Установление соответствия состава активов структурной и логической схемам
4. Проверка достаточности мер защиты:
   - Анализ локальных актов
   - Испытания системы защиты
   - Внешняя и внутренняя проверка уязвимостей
   - **Тестирование на проникновение** (с использованием сетевого сканера, сканера уязвимостей, сканера уязвимостей веб-приложений)
5. Оформление технического отчёта и протокола испытаний
6. Оформление аттестата соответствия

### 6.3. Срок действия аттестата

Аттестат соответствия оформляется сроком на **5 лет**.
