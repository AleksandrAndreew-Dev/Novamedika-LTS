## 1. ОБЩИЕ СВЕДЕНИЯ

### 1.1. Наименование информационной системы
**NovaMedika2** — информационная система поиска лекарственных средств, оформления заказов и проведения онлайн-консультаций между пользователями и фармацевтами.

### 1.2. Собственник (владелец) информационной системы
- **Наименование:** [НАИМЕНОВАНИЕ ОРГАНИЗАЦИИ / ИП]
- **Место нахождения:** [АДРЕС]
- **Регистрационный номер:** [НОМЕР В ЕГР]
- **Контактное лицо:** [ФИО, должность]
- **Контактный телефон:** [ТЕЛЕФОН]
- **Email:** [EMAIL]

### 1.3. Назначение информационной системы
Информационная система предназначена для:
- Поиска лекарственных средств по наименованию, действующему веществу, производителю
- Просмотра информации о наличии лекарств в аптеках-партнёрах
- Оформления бронирования (заказов) на лекарства
- Проведения онлайн-консультаций между пользователями и сертифицированными фармацевтами через Telegram-бот
- Синхронизации данных о наличии и ценах с аптеками-партнёрами

### 1.4. Категория обрабатываемой информации
В информационной системе обрабатывается следующая информация:
- **Персональные данные** (не специальные):
  - Пользователи: Telegram ID, имя, фамилия, номер телефона
  - Фармацевты: Telegram ID, ФИО, номер телефона, данные о квалификации
  - Заказчики: имя, номер телефона, данные о заказах
- **Коммерческая информация**: данные о лекарствах, цены, наличие в аптеках
- **Служебная информация**: логи аутентификации, журналы событий ИБ

**Не обрабатываются:**
- ❌ Специальные персональные данные (биометрические, генетические, о здоровье)
- ❌ Государственные секреты
- ❌ Служебная информация ограниченного распространения (гриф "ДСП")

### 2.2. Технические характеристики

| Параметр | Значение |
|----------|----------|
| **Тип архитектуры** | Клиент-серверная, микросервисная |
| **Платформа** | Linux (Ubuntu 22.04 LTS) |
| **Контейнеризация** | Docker + Docker Compose |
| **Веб-сервер** | Nginx (frontend), Traefik (reverse proxy) |
| **Backend фреймворк** | FastAPI (Python 3.11+) |
| **Frontend фреймворк** | React 18 + Vite |
| **СУБД** | PostgreSQL 15+ |
| **Кэш/очередь** | Redis 7+ |
| **Асинхронные задачи** | Celery + Redis Broker |
| **Протоколы связи** | HTTPS (TLS 1.2+), WSS (WebSocket Secure) |
| **Аутентификация** | JWT (access + refresh tokens) |
| **Шифрование данных** | AES-256 (pgcrypto для БД), Fernet (для токенов) |

### 2.3. Подключение к открытым каналам передачи данных

✅ **Информационная система подключена к открытым каналам передачи данных:**
- Интернет-канал для доступа пользователей к Web App (HTTPS)
- Telegram Bot API (через webhook, HTTPS)
- API внешних сервисов (Tabletka.by для синхронизации данных аптек)
- Let's Encrypt API для получения SSL-сертификатов

## 3. НОРМАТИВНАЯ ПРАВОВАЯ БАЗА

### 3.1. Основные нормативные документы

1. **Закон Республики Беларусь от 07.05.2021 № 99-З** "О защите персональных данных"
2. **Указ Президента Республики Беларусь от 04.10.2018 № 449** "О мерах по совершенствованию государственной системы защиты информации"
3. **Приказ ОАЦ от 20 февраля 2020 г. № 66** "Об утверждении Положения о порядке технической и криптографической защиты информации" **в редакции Приказа ОАЦ от 10 декабря 2024 г. № 259**
4. Закон Республики Беларусь от 07.05.2021 № 455-З "Об основах деятельности по профилактике правонарушений"

### 3.2. Анализ согласно Приложению 1 к Положению (редакция Приказа № 259)

Согласно **Приложению 1** к Положению о порядке технической и криптографической защиты информации (в редакции Приказа ОАЦ № 259), классы типовых информационных систем определяются следующими критериями:

| Критерий | Значение для NovaMedika2 |
|----------|---------------------------|
| **Тип обрабатываемой информации** | Персональные данные (не специальные) |
| **Подключение к открытым каналам** | ✅ Да (Интернет, Telegram API) |
| **Государственная/негосударственная** | Негосударственная |

### 3.3. Определение класса

На основании анализа:

1. **Обрабатываются персональные данные** (Telegram ID, имена, телефоны пользователей и фармацевтов)
2. **НЕ обрабатываются специальные персональные данные** (биометрические, генетические, о состоянии здоровья)
3. **Информационная система подключена к открытым каналам передачи данных** (Интернет, Telegram Bot API)

Согласно пункту 10 **Приложения 1** к Положению (в редакции Приказа ОАЦ № 259):

> **Класс 3-ин** – информационные системы, в которых обрабатываются персональные данные, за исключением специальных персональных данных, и которые подключены к открытым каналам передачи данных.

### 3.4. Обоснование

✅ **Информационная система NovaMedika2 относится к классу 3-ин**, так как:
- Обрабатывает персональные данные пользователей и фармацевтов
- Не обрабатывает специальные категории персональных данных
- Имеет подключение к открытым каналам передачи данных (Интернет, Telegram API)
- Является негосударственной информационной системой

## 4. ТРЕБОВАНИЯ К СИСТЕМЕ ЗАЩИТЫ ИНФОРМАЦИИ

Для информационной системы класса **3-ин** должны быть реализованы требования согласно **Приложению 3** к Положению (в редакции Приказа ОАЦ № 259):

### 4.1. Обязательные требования (+)

#### Раздел 1. Аудит безопасности
- ✅ 1.1 Регистрация событий ИБ (аутентификация, нарушения доступа)
- ✅ 1.2 Хранение логов не менее 1 года
- ✅ 1.3 Централизованный сбор логов (Docker logging + ELK/Prometheus)
- ✅ 1.4 Мониторинг событий ИБ уполномоченными лицами
- ✅ 1.5 Хранение информации о функционировании СВТ и сетевого оборудования

#### Раздел 2. Защита данных
- ✅ 2.1 Регламент использования съемных носителей
- ✅ 2.2 Контроль работоспособности средств защиты
- ✅ 2.3 Защита резервных копий и параметров настройки

#### Раздел 3. Идентификация и аутентификация
- ✅ 3.1 Разграничение доступа (роли: user, pharmacist, admin)
- ✅ 3.2 Идентификация и аутентификация пользователей (JWT)
- ⚠️ 3.3 Защита обратной связи при вводе пароля (masked input)
- ✅ 3.4 Управление учетными записями (создание, блокировка, удаление)
- ✅ 3.5 Контроль правил генерации и смены паролей
- ⚠️ 3.6 Централизованное управление учетными записями
- ✅ 3.7 Блокировка после неактивности (timeout сессии)

#### Раздел 4. Защита СЗИ
- ✅ 4.1 Изменение атрибутов безопасности по умолчанию
- ✅ 4.2 Обновление объектов ИС (security patches)
- ✅ 4.3 Контроль физического доступа (дата-центр)
- ✅ 4.4 Синхронизация времени (NTP)

#### Раздел 5. Криптографическая защита
- ✅ 5.1 Шифрование при передаче (HTTPS/TLS 1.2+)
- ⚠️ 5.2 Шифрование при хранении (AES-256 для чувствительных полей)
- ✅ 5.3 ЭЦП для электронных документов (при необходимости)
- ⚠️ 5.4 Контроль целостности данных (hash verification)
- ⚠️ 5.5 Защита личных ключей ЭЦП (криптографические токены)
- ➖ 5.6 Многофакторная аутентификация (опционально для класса 3-ин)

#### Раздел 6. Виртуальная инфраструктура
- ✅ 6.1 Защита от агрессивного использования ресурсов (Docker limits)
- ✅ 6.2 Защита виртуальной инфраструктуры от атак
- ✅ 6.3 Безопасное перемещение ВМ
- ✅ 6.4 Резервное копирование ВМ
- ➖ 6.5 Резервирование сетевого оборудования N+1 (опционально)
- ➖ 6.6 Физическая изоляция сегмента (опционально)

#### Раздел 7. Иные требования
- ✅ 7.1 Перечень разрешенного ПО
- ✅ 7.2 Контроль состава объектов ИС
- ⚠️ 7.3 Автоматизированный контроль СВТ и сетевого оборудования
- ✅ 7.4 Использование под пользовательскими учетными записями
- ✅ 7.5 Антивирусная защита
- ✅ 7.6 Межсетевое экранирование
- ✅ 7.7 Обнаружение и предотвращение вторжений (IDS/IPS)
- ✅ 7.8 Контроль подключений к открытым каналам
- ✅ 7.9 Защита от вредоносного кода
- ✅ 7.10 Обновление сигнатур антивируса
- ✅ 7.11 Сканирование уязвимостей
- ✅ 7.12 Локализация и ликвидация последствий нарушений

### 4.2. Требования с условным выполнением (+/–)

Следующие требования являются опциональными для класса 3-ин и могут не выполняться при обосновании:
- 1.3 Централизованный сбор логов (если нет централизованной системы)
- 3.3 Защита обратной связи (если ввод только через защищенные формы)
- 3.6 Централизованное управление учетными записями (если мало пользователей)
- 5.2, 5.4, 5.5 Криптозащита при хранении (зависит от категории данных)
- 5.6 Многофакторная аутентификация (рекомендуется, но не обязательна)
- 6.5, 6.6 Требования к виртуальной инфраструктуре (зависит от архитектуры)
- 7.3 Автоматизированный контроль оборудования
