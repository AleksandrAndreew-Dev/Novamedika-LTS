## 1. ОБЩИЕ ПОЛОЖЕНИЯ

1.1. Настоящая Политика информационной безопасности (далее — Политика) определяет цели, принципы и обязательства Организации по обеспечению защиты информации в информационной системе **NovaMedika2** (далее — ИС, Система).

1.2. ИС NovaMedika2 — справочный сервис поиска лекарственных средств, оформления заказов и проведения онлайн-консультаций между пользователями и фармацевтами.

1.3. Класс типовых информационных систем: **3-ин** (персональные данные + открытые каналы передачи данных).

1.4. Настоящая Политика обязательна для исполнения всеми работниками Организации, имеющими доступ к активам ИС NovaMedika2.

1.5. Политика доводится до сведения работников в части, их касающейся, и доступна заинтересованным субъектам информационных отношений для ознакомления.

## 2. ЦЕЛИ И ПРИНЦИПЫ ЗАЩИТЫ ИНФОРМАЦИИ

### 2.1. Цели

- Обеспечение конфиденциальности персональных данных пользователей и фармацевтов
- Обеспечение целостности и доступности данных ИС
- Предотвращение несанкционированного доступа к информации

- **Законность** — обработка информации в соответствии с законодательством
- **Минимальные привилегии** — доступ к информации предоставляется в минимально необходимом объёме
- **Непрерывность совершенствования** — постоянное улучшение системы защиты информации
- **Комплексность** — применение совокупности правовых, организационных и технических мер
- **Прозрачность** — информирование субъектов ПД о порядке обработки их данных
- **Сертификация средств защиты** — использование сертифицированных средств криптографической и технической защиты информации в соответствии с требованиями Указа №449 и Закона №455-З

## 4. СИСТЕМА ЗАЩИТЫ ИНФОРМАЦИИ

### 4.1. Состав системы защиты информации

Система защиты информации ИС NovaMedika2 включает совокупность средств и мер:

**Средства криптографической защиты:**
- TLS-шифрование (Let's Encrypt, ACME) для защиты передаваемых данных
- JWT-аутентификация (access token 30 мин, refresh token 7 дней)
- Fernet-шифрование для защиты чувствительных данных в БД (токены API аптек)
- AES-256 шифрование через pgcrypto для персональных данных (телефоны, Telegram ID)

**Важное примечание о сертификации:** Согласно п.13 Положения о технической и криптографической защите информации (Указ №449) и п.7 ст.28 Закона №455-З, средства технической и криптографической защиты должны иметь сертификат соответствия Национальной системы подтверждения соответствия или положительное экспертное заключение ОАЦ. В настоящее время используемые библиотеки (cryptography/Fernet, PyJWT, pgcrypto) не имеют таких сертификатов. Данный вопрос находится в стадии проработки — рассматриваются варианты замены на сертифицированные аналоги, получения экспертного заключения или согласования компенсирующих мер с ОАЦ.

**Средства защиты от НСД:**
- Аутентификация: JWT Bearer, HTTP Basic Auth, API Keys, Redis AUTH, PostgreSQL AUTH
- Разграничение доступа: роли user / pharmacist / admin
- Docker non-root — все контейнеры от непривилегированных пользователей
- Docker resource limits — изоляция ресурсов контейнеров

**Средства контроля целостности:**
- HTTPS/TLS 1.3 для контроля целостности передаваемых данных (SSL Labs A+)
- Health checks (Docker HEALTHCHECK, Traefik health endpoint)
- Database constraints (UNIQUE, FOREIGN KEY, CHECK) для контроля целостности данных
- SHA-256 checksums для резервных копий (scripts/backup.sh)
- Мониторинг событий ИБ (см. [07-ib-monitoring-reglament.md](07-ib-monitoring-reglament.md))

**Средства межсетевого экранирования:**
- Traefik v3.6 — обратный прокси с маршрутизацией и TLS-терминацией
- Docker bridge network — сегментация внутренней сети
- Security headers: CSP (frame-ancestors 'self' https://t.me https://web.telegram.org), HSTS (max-age=31536000, includeSubDomains, preload), X-Frame-Options=SAMEORIGIN, XSS-Filter, Referrer-Policy, Permissions-Policy
- BasicAuth для Traefik dashboard (защита служебного интерфейса)
- Slowapi rate limiting на backend (защита от brute-force)
- CORS policy с whitelist origins

**Средства мониторинга и аудита:**
- **Audit middleware** (ackend/src/middleware/audit_middleware.py) — логирование обращений к ПД в таблицу udit_logs (требование ОАЦ п.2.1)
- **Prometheus metrics** (/metrics endpoint) — REQUEST_COUNT, REQUEST_LATENCY, ACTIVE_REQUESTS (требование ОАЦ п.1.5)
- Traefik Prometheus exporter (метрики инфраструктуры)
- Health checks для всех контейнеров (Docker HEALTHCHECK, интервал 30s)
- Логи контейнеров в JSON-формате (json-file driver, ротация max-size=10m, max-file=3)

**Средства защиты от НСД (дополнение):**
- Двухэтапная аутентификация фармацевтов: (1) Telegram ID + (2) REGISTRATION_SECRET_WORD
- Маскирование полей ввода паролей (type=password, autocomplete=off)
- Session timeout 30 мин для веб-приложения (React hook useSessionTimeout)
- FSM timeout 15 мин для Telegram-бота
- IP whitelist / rate limiting для админ-эндпоинтов

## 6. ПОРЯДОК ПРИМЕНЕНИЯ СИСТЕМЫ ЗАЩИТЫ ИНФОРМАЦИИ

### 6.1. Выявление угроз и реагирование на события ИБ

| Тип события | Действия | Ответственный | Срок |
|------------|----------|--------------|------|
| Несанкционированный доступ | Блокировка учётной записи, расследование | Администратор | Немедленно |
| Сбой функционирования ИС | Восстановление из резервной копии | Администратор | В течение 4 часов |
| Утечка персональных данных | Информирование ОАЦ и Национального центра защиты персональных данных (НЦЗПД) | Ответственный за защиту | До 3 рабочих дней (в соответствии с п.7 ст.16 Закона №99-З) |
| Компрометация криптографических ключей | Прекращение использования, перевыпуск | Администратор | Немедленно |
| Нарушение политики ИБ | Расследование, устранение нарушений | Ответственный за защиту | В течение 5 рабочих дней |

### 6.2. Порядок управления учётными записями (детализировано)

- **Создание учётной записи фармацевта:** через Telegram-бот с двухэтапной верификацией (Telegram ID + REGISTRATION_SECRET_WORD)
- **Создание учётной записи администратора:** по приказу руководителя Организации
- **Аутентификация:** JWT (access token 30 мин + refresh token 7 дней с rotation и отзыванием в БД), bcrypt для хэширования паролей
- **Блокировка:** при увольнении, нарушении правил, неактивности > 90 дней, а также автоматически при попытках brute-force (slowapi)
- **Удаление:** по истечении 1 года после блокировки, либо по запросу субъекта ПД (GDPR-compliant API endpoints)
- **Контроль паролей:** API-ключи аптек — минимум 16 символов, генерация через secrets.token_urlsafe(32). Пользовательские пароли — минимум 8 символов, принудительная смена каждые 90 дней (запланировано Q2 2026)

- Создание учётной записи — по приказу руководителя или через процедуру регистрации
- Блокировка — при увольнении, нарушении правил, неактивности более 90 дней
- Удаление — по истечении 1 года после блокировки
- Контроль паролей: минимальная длина 12 символов, регулярная смена

### 6.3. Порядок обновления программного обеспечения

- Обновление Docker-образов — через CI/CD pipeline (GitHub Actions)
- Обновление зависимостей Python: ручной контроль (uv.lock) + Dependabot для security-обновлений
- Обновление зависимостей JS: Dependabot для npm (package-lock.json)
- Тестирование обновлений перед внедрением в production
- Контроль своевременности — ответственный администратор

### 6.4. Порядок резервного копирования

| Объект | Метод | Периодичность | Хранение |
|--------|-------|--------------|----------|
| PostgreSQL БД | pg_dump | Ежедневно | 30 дней |
| Конфигурации (docker-compose, nginx, traefik) | Копирование | При изменении | Бессрочно |
| TLS-сертификаты | Автоматически (Let's Encrypt) | При обновлении | — |
| Логи контейнеров | Docker json-file | Непрерывно | 365 дней |

### 6.5. Синхронизация системного времени

Системное время всех контейнеров синхронизируется от единого источника (NTP на хост-сервере).

### 8.3. Внешние сервисы и ссылки для пользователей

8.3.1. Информационная система NovaMedika2 содержит ссылки на внешние ресурсы, которые открываются по инициативе пользователя:

| Внешний сервис | Назначение | Передаваемые данные | Способ открытия |
|---------------|------------|---------------------|-----------------|
| Google Maps / Yandex Maps / OpenStreetMap | Отображение расположения аптек на карте | Адрес аптеки (публичные данные) | Новая вкладка браузера (target="_blank", rel="noopener noreferrer") |

8.3.2. Особенности переходов на внешние ресурсы:

- Переходы осуществляются исключительно по явной инициативе пользователя (клик по ссылке/кнопке)
- Внешние ссылки открываются в новой вкладке браузера с атрибутами безопасности `rel="noopener noreferrer"`
- Атрибут `noopener` предотвращает доступ внешнего сайта к объекту `window.opener` нашего приложения
- Атрибут `noreferrer` предотвращает передачу HTTP-заголовка Referer, защищая историю переходов
- На внешние ресурсы передаются только публичные данные (адреса аптек), персональные данные пользователей НЕ передаются

8.3.3. Внешние сервисы НЕ получают доступ к:

- Персональным данным пользователей (имена, телефоны, email, Telegram ID)
- Данным заказов и бронирований
- Историям консультаций и диалогов
- Данным фармацевтов
- Конфигурационным данным системы

8.3.4. Ответственность за обработку данных на внешних сервисах:

- Оператор не несёт ответственность за политику конфиденциальности внешних сервисов
- Пользователи уведомляются о переходе на внешний ресурс визуальной индикацией (иконка ↗)
- Рекомендуется ознакомиться с политикой конфиденциальности внешнего сервиса перед использованием
- Внешние сервисы обрабатывают данные в соответствии со своими политиками и законодательством своих юрисдикций

8.3.5. Технические меры безопасности:

- Все внешние URL валидируются перед открытием (проверка домена)
- Разрешены только доверенные картографические сервисы (google.com, yandex.ru, openstreetmap.org)
- Запрещено открытие внешних ссылок через серверный прокси (переход происходит на стороне клиента)
- Логирование фактов использования функции карты осуществляется без передачи деталей во внешние системы

## 9. АТТЕСТАЦИЯ СИСТЕМЫ ЗАЩИТЫ ИНФОРМАЦИИ

9.1. Система защиты информации ИС NovaMedika2 подлежит обязательной аттестации в соответствии с Положением о порядке аттестации систем защиты информации (Приказ ОАЦ №66).

9.2. Аттестация проводится до ввода ИС в эксплуатацию или после внесения изменений в систему защиты информации.

9.3. Процедура аттестации включает:
- Разработку программы и методики испытаний
- Проведение испытаний системы защиты
- Оформление технического отчёта
- Получение аттестата соответствия сроком действия 5 лет

9.4. Эксплуатация ИС с ограниченным доступом без действующего аттестата соответствия запрещена.

9.5. Ответственность за организацию и проведение аттестации возлагается на ответственного за обеспечение защиты информации.
