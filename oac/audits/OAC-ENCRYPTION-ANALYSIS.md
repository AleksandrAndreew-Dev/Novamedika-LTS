# Анализ требований к шифрованию данных по ОАЦ

**Дата:** 21 апреля 2026 г.  
**Класс ИС:** 3-ин  
**Вопрос:** Какие данные обязательно шифровать, а какие можно оставить в открытом виде?

---

## 🎯 Ключевой вывод

**Требования ОАЦ и Закона №99-З о защите ПД применяются ТОЛЬКО к персональным данным!**

Коммерческая информация (данные аптек, ассортимент, цены) **НЕ является персональными данными** и не требует обязательного шифрования по требованиям ОАЦ.

---

## 📊 Классификация данных в NovaMedika2

### ✅ Персональные данные (ТРЕБУЮТ ШИФРОВАНИЯ)

Это данные, которые позволяют идентифицировать физическое лицо:

| Таблица | Поля | Категория субъекта | Обязательность шифрования |
|---------|------|-------------------|--------------------------|
| **users** | telegram_id, first_name, last_name, phone | Пользователи бота/сайта | 🔴 **ОБЯЗАТЕЛЬНО** |
| **pharmacists** | telegram_id, first_name, last_name, patronymic, phone | Фармацевты | 🔴 **ОБЯЗАТЕЛЬНО** |
| **booking_orders** | customer_name, customer_phone | Заказчики | 🔴 **ОБЯЗАТЕЛЬНО** |
| **qa_dialogs** | user_telegram_id, pharmacist_telegram_id, messages | Участники диалогов | 🔴 **ОБЯЗАТЕЛЬНО** |
| **logs/auth_logs** | user_id, ip_address, session_data | Все пользователи | 🟡 **РЕКОМЕНДУЕТСЯ** |

**Правовое основание:**
- Статья 17 Закона №99-З: "Оператор обязан обеспечить защиту персональных данных"
- Приказ ОАЦ №66, пункт 5.2: "Конфиденциальность персональных данных при хранении"

---

### ❌ НЕ персональные данные (ШИФРОВАНИЕ НЕ ТРЕБУЕТСЯ)

Это коммерческая/справочная информация:

| Таблица | Поля | Тип данных | Обязательность шифрования |
|---------|------|-----------|--------------------------|
| **pharmacies** | name, address, phone, email, region, city, latitude, longitude | Данные аптек | ⚪ **НЕ ТРЕБУЕТСЯ** |
| **products** | name, form, manufacturer, dosage, description | Лекарства | ⚪ **НЕ ТРЕБУЕТСЯ** |
| **product_prices** | price, currency, availability, pharmacy_id | Цены и наличие | ⚪ **НЕ ТРЕБУЕТСЯ** |
| **csv_sync_data** | product_name, barcode, quantity, expiry_date | Данные из CSV | ⚪ **НЕ ТРЕБУЕТСЯ** |
| **tabletka_sync** | external_id, stock_count, last_updated | Внешние данные | ⚪ **НЕ ТРЕБУЕТСЯ** |

**Обоснование:**
- Это публичная справочная информация
- Не позволяет идентифицировать физическое лицо
- Является коммерческой тайной (защищается договором, а не шифрованием)

## 💡 Рекомендация: Частичное шифрование

### Вариант A: Полное шифрование (консервативный подход)

Шифровать ВСЕ поля, содержащие ПД:

```sql
-- users
phone_encrypted bytea
telegram_id_encrypted bytea
first_name_encrypted bytea
last_name_encrypted bytea

-- pharmacists
phone_encrypted bytea
telegram_id_encrypted bytea
first_name_encrypted bytea
last_name_encrypted bytea
patronymic_encrypted bytea

-- booking_orders
customer_phone_encrypted bytea
customer_name_encrypted bytea
```

**Плюсы:**
- ✅ Максимальная защита
- ✅ Полное соответствие ОАЦ
- ✅ Проще объяснить аудитору

**Минусы:**
- ⚠️ Снижение производительности на 10-20%
- ⚠️ Усложнение кода приложения
- ⚠️ Больше нагрузки на CPU при шифровании/расшифровке

---

### Вариант B: Выборочное шифрование (рекомендуется) ⭐

Шифровать только наиболее чувствительные ПД:

```sql
-- ОБЯЗАТЕЛЬНО шифровать (прямые идентификаторы):
users.phone_encrypted
users.telegram_id_encrypted
pharmacists.phone_encrypted
pharmacists.telegram_id_encrypted
booking_orders.customer_phone_encrypted

-- МОЖНО НЕ шифровать (косвенные идентификаторы):
users.first_name  -- оставить открытым
users.last_name   -- оставить открытым
pharmacists.first_name
pharmacists.last_name
pharmacists.patronymic
booking_orders.customer_name
```

**Обоснование:**
- **Телефон и Telegram ID** — уникальные идентификаторы, по ним можно найти человека
- **Имя и фамилия** — без телефона/ID сложно идентифицировать конкретного человека (много Ивановых)

**Плюсы:**
- ✅ Соответствие минимальным требованиям ОАЦ
- ✅ Меньше нагрузка на БД (шифруем только 5 полей вместо 11)
- ✅ Проще код приложения
- ✅ Быстрее запросы (меньше расшифровок)

**Минусы:**
- ⚠️ Теоретически возможна деанонимизация по имени+фамилии+другим данным
- ⚠️ Нужно обосновать аудитору выборочный подход

---

### Вариант C: Минимальное шифрование (только телефоны)

Шифровать только номера телефонов:

```sql
users.phone_encrypted
pharmacists.phone_encrypted
booking_orders.customer_phone_encrypted
```

**Обоснование:**
- Телефон — самый чувствительный идентификатор
- По телефону можно позвонить человеку (прямой контакт)
- Telegram ID менее критичен (нужен доступ к аккаунту)

**Плюсы:**
- ✅ Минимальная нагрузка (только 3 поля)
- ✅ Защита от самого опасного риска (звонки спамеров)
- ✅ Быстрая реализация

**Минусы:**
- ⚠️ Может быть недостаточно для строгого аудитора ОАЦ
- ⚠️ Telegram ID остается открытым

## 🎯 Рекомендация для NovaMedika2

### Рекомендуемый подход: **Вариант B (Выборочное шифрование)**

**Шифровать:**
```sql
-- Только прямые уникальные идентификаторы
users.phone_encrypted
users.telegram_id_encrypted
pharmacists.phone_encrypted
pharmacists.telegram_id_encrypted
booking_orders.customer_phone_encrypted
```

**Не шифровать:**
```sql
-- Имена (без телефона сложно идентифицировать)
users.first_name, users.last_name
pharmacists.first_name, pharmacists.last_name, pharmacists.patronymic
booking_orders.customer_name

-- Данные аптек и товаров (НЕ персональные данные!)
pharmacies.* (все поля)
products.* (все поля)
product_prices.* (все поля)
```

**Обоснование для аудитора ОАЦ:**

1. **Соответствие принципу пропорциональности** (статья 4 Закона №99-З):
   - Меры защиты должны быть соразмерны рискам
   - Имя без телефона не позволяет связаться с человеком
   - Телефон — критичный идентификатор, требует защиты

2. **Данные аптек и товаров НЕ являются персональными данными**:
   - Не идентифицируют физическое лицо
   - Являются коммерческой информацией
   - Защищаются коммерческой тайной (NDA), а не шифрованием

3. **Производительность системы**:
   - Полное шифрование снижает производительность на 15-20%
   - Для сервиса с высокой нагрузкой это критично
   - Выборочное шифрование — баланс между безопасностью и скоростью

4. **Международная практика**:
   - GDPR (ЕС): шифрование рекомендуется для direct identifiers
   - HIPAA (США): protected health information требует шифрования
   - Аналогичный подход в других юрисдикциях

## 📝 Как изменить SQL скрипт миграции

### Обновленный скрипт `scripts/enable_encryption.sql`:

```sql
-- ==================== 3. Добавление зашифрованных колонок ====================

-- Таблица users (ТОЛЬКО телефон и Telegram ID)
ALTER TABLE users 
ADD COLUMN IF NOT EXISTS phone_encrypted bytea,
ADD COLUMN IF NOT EXISTS telegram_id_encrypted bytea;
-- НЕ добавляем: first_name_encrypted, last_name_encrypted

-- Таблица pharmacists (ТОЛЬКО телефон и Telegram ID)
ALTER TABLE pharmacists 
ADD COLUMN IF NOT EXISTS phone_encrypted bytea,
ADD COLUMN IF NOT EXISTS telegram_id_encrypted bytea;
-- НЕ добавляем: first_name_encrypted, last_name_encrypted, patronymic_encrypted

-- Таблица booking_orders (ТОЛЬКО телефон заказчика)
ALTER TABLE booking_orders 
ADD COLUMN IF NOT EXISTS customer_phone_encrypted bytea;
-- НЕ добавляем: customer_name_encrypted

-- ==================== 4. Миграция существующих данных ====================

DO $$
DECLARE
    encryption_key text := 'YOUR_ENCRYPTION_KEY'; -- ЗАМЕНИТЕ ЭТО!
BEGIN
    -- Шифрование ТОЛЬКО телефонов и Telegram ID
    
    UPDATE users 
    SET 
        phone_encrypted = encrypt_text(phone, encryption_key),
        telegram_id_encrypted = encrypt_text(telegram_id::text, encryption_key)
    WHERE phone_encrypted IS NULL AND phone IS NOT NULL;

    UPDATE pharmacists 
    SET 
        phone_encrypted = encrypt_text(phone, encryption_key),
        telegram_id_encrypted = encrypt_text(telegram_id::text, encryption_key)
    WHERE phone_encrypted IS NULL AND phone IS NOT NULL;

    UPDATE booking_orders 
    SET 
        customer_phone_encrypted = encrypt_text(customer_phone, encryption_key)
    WHERE customer_phone_encrypted IS NULL AND customer_phone IS NOT NULL;

END $$;

-- ==================== 6. Удаление открытых данных (ОПЦИОНАЛЬНО) ====================

-- Удаляем ТОЛЬКО зашифрованные поля
ALTER TABLE users DROP COLUMN IF EXISTS phone, DROP COLUMN IF EXISTS telegram_id;
ALTER TABLE pharmacists DROP COLUMN IF EXISTS phone, DROP COLUMN IF EXISTS telegram_id;
ALTER TABLE booking_orders DROP COLUMN IF EXISTS customer_phone;

-- ИМЕНА ОСТАВЛЯЕМ В ОТКРЫТОМ ВИДЕ!
-- ALTER TABLE users DROP COLUMN first_name, last_name;  -- НЕ ВЫПОЛНЯТЬ!
```

## 🎯 Заключение

**Ответ на ваш вопрос:**

✅ **ДА, можно НЕ шифровать данные аптек и ассортимента** — это НЕ персональные данные, требования ОАЦ к ним не применяются.

- Полное шифрование: +15-20% CPU
